Ця лабораторна робота є **продовженням** роботи "CI/CD з безпековими перевірками", де ви замінювали test_app.py на версію з вразливостями та спостерігали за "Fail Fast" механізмом.

## 🔍 Хід роботи

### Крок 1: Ідентифікація зміни

#### 1.1 Знайдіть коміт із змінами test_app.py

**Через GitLab UI:**

1. Відкрийте ваш GitLab проєкт
2. Перейдіть: **Repository** → **Commits**
3. Знайдіть коміт з повідомленням про зміну test_app.py
   - Наприклад: `"Add vulnerable code for testing"` або `"Replace test_app.py with Bad version"`

**Або через історію файлу:**

1. **Repository** → **Files**
2. Знайдіть файл `test_app.py`
3. Натисніть **History** (іконка годинника)
4. Перегляньте всі зміни файлу

**Скриншот для звіту:** Зробіть скриншот списку комітів з виділеним цільовим комітом.

#### 1.2 Визначте основні атрибути коміту

**Відкрийте деталі коміту та зафіксуйте:**

| Атрибут | Де знайти | Приклад |
|---------|-----------|---------|
| **Commit SHA** | Унікальний ідентифікатор | `a3f5b2c7d8e9f...` |
| **Автор** | Author field | `Yevheniia T. <email@example.com>` |
| **Дата та час** | Commit date | `2025-12-15 14:32:18` |
| **Повідомлення коміту** | Commit message | `"Add vulnerable code for testing"` |
| **Гілка** | Branch name | `main` або `feature-branch` |
| **Кількість змін** | Files changed | `1 file changed, +150, -80` |

**Для звіту запишіть:**

```
ІДЕНТИФІКАЦІЯ КОМІТУ:

Commit SHA: a3f5b2c7d8e9f1234567890abcdef
Автор: Yevheniia T. <yevheniiat@example.com>
Дата: 2025-12-15 14:32:18 UTC
Повідомлення: "Add vulnerable code for testing"
Гілка: main
Файли змінені: 1 (test_app.py)
Додано рядків: +150
Видалено рядків: -80
```

#### 1.3 Перевірте цифровий підпис коміту

**Як перевірити в GitLab UI:**

1. Відкрийте деталі коміту
2. Шукайте позначку біля імені автора:
   - ✅ **Verified** (зелена галочка) - коміт підписаний та перевірений
   - 🔓 **Unverified** (сірий значок) - коміт НЕ підписаний
   - ❌ **Verification failed** - підпис невалідний

**Типи підписів:**

- **GPG Signature** - підпис GPG ключем
- **SSH Signature** - підпис SSH ключем
- **X.509 Signature** - сертифікат

**Через Git CLI (опціонально):**

```bash
# Клонуйте репозиторій
git clone <your-gitlab-repo-url>
cd <repo-name>

# Перевірте підпис коміту
git log --show-signature -1 <commit-sha>

# Або детальніше
git verify-commit <commit-sha>
```

**Для звіту:**

```
ПЕРЕВІРКА ЦИФРОВОГО ПІДПИСУ:

Статус підпису: ❌ Unverified (коміт не підписаний)
Тип підпису: Відсутній
GPG Key ID: N/A
Email підпису: N/A

⚠️ ПОРУШЕННЯ: Коміт не має цифрового підпису, що суперечить
політикам безпеки для критичних змін у коді.
```

#### 1.4 Визначте метод внесення зміни

**Перевірте, як була внесена зміна:**

**Варіант A: Прямий коміт у main (найгірший сценарій)**
```
Developer → [commit] → main
          (без review, без перевірок)
```

**Варіант B: Через Merge Request (правильний процес)**
```
Developer → [feature branch] → [MR + review] → main
```

**Як перевірити:**

1. Відкрийте коміт у GitLab
2. Подивіться на розділ **Related Merge Requests**
   - Якщо порожньо → прямий коміт у main ❌
   - Якщо є MR → зміна через merge request ✅

3. Або перевірте через CLI:
```bash
# Подивіться, чи був коміт частиною MR
git log --oneline --graph --all --decorate | grep <commit-sha>
```

**Для звіту:**

```
МЕТОД ВНЕСЕННЯ ЗМІНИ:

Тип зміни: ❌ Прямий коміт у гілку main
Merge Request: Відсутній
Code Review: Не проводився
Гілка розробки: Відсутня (коміт безпосередньо в main)

⚠️ ПОРУШЕННЯ: Зміна внесена напряму в main без створення
окремої гілки, без code review та без approval процесу.
```

### Крок 2: Аналіз змісту зміни

#### 2.1 Перегляньте diff (різницю) зміни

**Через GitLab UI:**

1. Відкрийте деталі коміту
2. Перегляньте вкладку **Changes**
3. Проаналізуйте червоні (-) та зелені (+) рядки

**Основні питання для аналізу:**

| Питання | Що шукати | Приклад |
|---------|-----------|---------|
| Які функції додано? | Нові `def` блоки | `test_dangerous_eval()` |
| Які бібліотеки імпортовано? | `import` statements | `import subprocess, pickle` |
| Чи є жорстко закодовані секрети? | Паролі, API ключі | `API_KEY = "sk-123..."` |
| Чи використовуються небезпечні функції? | `eval()`, `exec()`, `subprocess.call()` | ✅ Присутні |
| Чи є SQL запити? | SQL statements | Немає прямих |
| Чи змінено критичну логіку? | Business logic | Тести стали небезпечними |

### Крок 3: Оцінка відповідності політикам SCM

#### 3.1 Перевірка підпису коміту

**Політика:** Всі коміти в main гілку ПОВИННІ бути підписані GPG/SSH ключем

| Критерій | Вимога | Фактичний стан | Статус |
|----------|--------|----------------|--------|
| Цифровий підпис | Обов'язковий | Відсутній | ❌ FAIL |
| Тип підпису | GPG або SSH | N/A | ❌ FAIL |
| Верифікація | Автоматична | Не виконувалась | ❌ FAIL |

**Для звіту:**

```
ПОЛІТИКА 1: Обов'язкові підписи комітів
───────────────────────────────────────────
Статус: ❌ ПОРУШЕНО

Деталі:
- Коміт не має цифрового підпису
- Не можливо підтвердити справжність автора
- Ризик: можлива підробка коміту

Рекомендація:
- Увімкнути обов'язкові GPG підписи на рівні проєкту
- Налаштувати відхилення непідписаних комітів
```

#### 3.2 Перевірка процесу Code Review

**Політика:** Всі зміни в main ПОВИННІ пройти code review (мінімум 1 approval)

| Критерій | Вимога | Фактичний стан | Статус |
|----------|--------|----------------|--------|
| Merge Request | Обов'язковий | Відсутній | ❌ FAIL |
| Code Review | Мін. 1 reviewer | Не проводився | ❌ FAIL |
| Approval | Required | Відсутній | ❌ FAIL |

**Для звіту:**

```
ПОЛІТИКА 2: Обов'язковий Code Review
──────────────────────────────────────
Статус: ❌ ПОРУШЕНО

Деталі:
- Коміт внесено напряму в main без Merge Request
- Code review не проводився
- Approval відсутній
- Ризик: небезпечний код потрапив у production без перевірки

Рекомендація:
- Заборонити прямі коміти в main (branch protection)
- Налаштувати обов'язковий approval для MR
- Мінімум 2 reviewers для security-критичних файлів
```

#### 3.3 Перевірка Branch Protection Rules

**Політика:** Main гілка повинна мати захист від прямих комітів

**Перевірте через GitLab:**
1. **Settings** → **Repository** → **Protected branches**
2. Перегляньте налаштування для `main` гілки

**Типові налаштування:**

| Правило | Рекомендовано | Фактично | Статус |
|---------|---------------|----------|--------|
| **Allowed to push** | No one | Maintainers | ❌ FAIL |
| **Allowed to merge** | Maintainers | Maintainers | ✅ PASS |
| **Require approval** | Yes (min 2) | No | ❌ FAIL |
| **Code owner approval** | Yes | No | ❌ FAIL |
| **Require signed commits** | Yes | No | ❌ FAIL |

**Для звіту:**

```
ПОЛІТИКА 3: Branch Protection Rules
────────────────────────────────────────
Статус: ⚠️ ЧАСТКОВЕ ПОРУШЕННЯ

Поточні налаштування main гілки:
┌─────────────────────────────┬──────────┬──────────┐
│ Правило                     │ Потрібно │ Факт     │
├─────────────────────────────┼──────────┼──────────┤
│ Заборона прямого push       │ ✅       │ ❌       │
│ Обов'язковий MR             │ ✅       │ ❌       │
│ Мінімум approvals           │ 2        │ 0        │
│ Підписані коміти            │ ✅       │ ❌       │
│ Status checks (CI)          │ ✅       │ ⚠️       │
│ CODEOWNERS review           │ ✅       │ ❌       │
└─────────────────────────────┴──────────┴──────────┘

Ризик: Main гілка недостатньо захищена, що дозволило
внести небезпечну зміну без належної перевірки.
```

**Перевірте:**
- Чи є файл `CHANGELOG.md` у репозиторії
- Чи згадується ця зміна в журналі

**Для звіту:**


### Крок 4: Оцінка ризиків та наслідків

#### 4.1 Матриця ризиків

**Створіть таблицю ризиків:**

```
МАТРИЦЯ РИЗИКІВ

┌────────────────────────────┬────────────┬──────────┬─────────────┐
│ Ризик                      │ Ймовірність│ Вплив    │ Пріоритет   │
├────────────────────────────┼────────────┼──────────┼─────────────┤
│ Code Execution             │ ВИСОКИЙ    │ КРИТИЧНИЙ│ P0 CRITICAL │
│ (eval/exec/subprocess)     │            │          │             │
├────────────────────────────┼────────────┼──────────┼─────────────┤
│ Hardcoded Secrets Leak     │ ВИСОКИЙ    │ ВИСОКИЙ  │ P1 HIGH     │
│ (API keys в коді)          │            │          │             │
├────────────────────────────┼────────────┼──────────┼─────────────┤
│ Unsafe Deserialization     │ СЕРЕДНІЙ   │ ВИСОКИЙ  │ P1 HIGH     │
│ (pickle.loads)             │            │          │             │
├────────────────────────────┼────────────┼──────────┼─────────────┤
│ Відсутність Code Review    │ ВИСОКИЙ    │ ВИСОКИЙ  │ P1 HIGH     │
│ (процесні проблеми)        │            │          │             │
├────────────────────────────┼────────────┼──────────┼─────────────┤
│ Непідписаний коміт         │ СЕРЕДНІЙ   │ СЕРЕДНІЙ │ P2 MEDIUM   │
│ (authenticity concerns)    │            │          │             │
├────────────────────────────┼────────────┼──────────┼─────────────┤
│ Weak Crypto (MD5)          │ НИЗЬКИЙ    │ СЕРЕДНІЙ │ P3 LOW      │
│                            │            │          │             │
└────────────────────────────┴────────────┴──────────┴─────────────┘

ЛЕГЕНДА:
Пріоритет = Ймовірність × Вплив
P0 = Критичний (негайна дія)
P1 = Високий (термінова дія < 24 год)
P2 = Середній (планова дія < тиждень)
P3 = Низький (backlog)
```

#### 4.2 Сценарії наслідків

**Можливі сценарії розвитку подій:**

```
СЦЕНАРІЙ 1: Експлуатація Code Execution
─────────────────────────────────────────
Якщо код потрапить у production:

1. Атакуючий знаходить endpoint, що викликає test функції
2. Передає malicious input у test_system_command_execution()
3. Виконується довільна команда на сервері
4. Можливий RCE (Remote Code Execution)

НАСЛІДКИ:
💥 Повний контроль над сервером
💥 Витік даних клієнтів
💥 Компрометація інфраструктури
💰 Фінансові збитки: $100K - $1M+
📉 Репутаційні втрати: критичні

═══════════════════════════════════════════

СЦЕНАРІЙ 2: Витік Hardcoded Secrets
─────────────────────────────────────
Якщо секрети потраплять у публічний репозиторій:

1. Git history містить API ключі
2. Сканери (GitGuardian, TruffleHog) їх знаходять
3. Атакуючі отримують доступ до API
4. Можливе несанкціоноване використання

НАСЛІДКИ:
💥 Компрометація AWS інфраструктури
💥 Несанкціонований доступ до БД
💥 Фінансові збитки за AWS використання
💰 Можливі витрати: $10K - $100K
📉 Необхідність ротації всіх ключів

═══════════════════════════════════════════

СЦЕНАРІЙ 3: Supply Chain Attack
─────────────────────────────────
Якщо коміт є частиною цілеспрямованої атаки:

1. Зловмисник отримав доступ до акаунта розробника
2. Внесено backdoor через test_app.py
3. Відсутність review дозволила проникнути в main
4. Код потрапляє в production builds

НАСЛІДКИ:
💥 Backdoor у production системі
💥 Можливість подальших атак
💥 Компрометація supply chain
💰 Збитки: $1M+
📉 Втрата довіри клієнтів
```

## 💡 Поради для виконання

### 1. Будьте систематичними
- Перевіряйте кожен пункт послідовно
- Документуйте ВСІ знахідки
- Робіть скриншоти одразу

### 2. Думайте як Security Engineer
- Не просто констатуйте факти
- Оцінюйте можливі наслідки
- Пропонуйте конкретні рішення

### 3. Використовуйте контекст
- Пам'ятайте про результати з попередньої лаби
- Зв'язуйте технічні факти з бізнес-ризиками
- Думайте про реальний вплив

### 4. Готуйтеся до ДЗ
- Зберігайте ВСІ знахідки
- Робіть детальні нотатки
- Скриншоти - ваші докази
